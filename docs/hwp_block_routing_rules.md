# HWP Block Routing Rules

## 목적

HWP에서 추출한 블록을 OCR 전처리 관점에서 어디로 보낼지 정리한다.

최종 목표는 세 가지다.

- RAG에 바로 넣을 수 있는 구조 데이터표를 남긴다
- 표 자체는 아니지만 뒤 표를 설명하는 제목 블록은 연결 메타데이터로 보존한다
- 설명문/라벨/조각 블록은 불필요하게 살리지 않는다

## 최종 라우팅

### 1. structural_table

실제 값과 항목 관계가 있는 표다.

예:

- 요구사항 표
- 평가표
- 일정/금액 표
- 제출서류 표
- 필드형 입력항목 표

처리:

- OCR 후보로 선별 가능
- RAG에 표 데이터로 적재

### 2. section_header_block

자체 데이터는 약하지만, 뒤 구조표의 상위 제목 역할을 하는 블록이다.

예:

- `가 / 기능 요구사항(System Function Requirement)`
- `제안 요구사항 / 요구사항 정의표`

처리:

- 단독 표로 적재하지 않음
- 뒤 구조표에 `linked_parent_text` 같은 메타데이터로 연결

### 3. explanatory_block

표라기보다 설명문/규정문 성격이 강한 블록이다.

예:

- 입찰 참가자격
- 입찰시 유의사항
- 제안서 작성방법
- 과업수행방법

처리:

- 표 OCR 대상 아님
- 필요하면 일반 텍스트 문맥으로만 사용

### 4. discarded

연결도 안 되고, 설명문 가치도 낮고, 데이터도 없는 블록이다.

예:

- 섹션 제목 조각
- 메뉴/라벨 조각
- 값 없는 빈 양식 조각

처리:

- 버림

## 판단 기준

### 구조 데이터표로 보는 기준

- 값/항목 관계가 있다
- 행 반복이 있다
- 헤더나 필드 그룹이 보인다
- RAG 질의응답에 쓸 수 있는 정형 정보가 있다

### 연결 제목 블록으로 보는 기준

- 길이가 짧다
- 제목/카테고리 성격이 강하다
- 바로 뒤에 구조표가 이어진다
- 단독 데이터는 약하지만 문맥 가치가 있다

### 설명문으로 보는 기준

- bullet와 긴 문장이 중심이다
- 규정/유의사항/방법론 설명이 많다
- 필드-값 구조가 없다

### 버리는 기준

- 값이 없다
- 뒤 표와 연결도 없다
- 설명문으로도 가치가 낮다
- 단순 라벨/장식/조각 수준이다

## 현재 해석 예시

- `table_index 42`
  - `section_header_block`
  - 뒤 `43~47` 요구사항 표에 연결

- `table_index 36`
  - `explanatory_block` 성격
  - 구조 데이터표로는 보지 않음

- `table_index 21`
  - 값이 없는 라벨 블록
  - 연결 관계가 없으면 버림

- `table_index 204`
  - 빈 양식 조각
  - 구조표로 살리지 않음
