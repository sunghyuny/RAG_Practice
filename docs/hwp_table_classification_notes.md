# HWP Table Classification Notes

## 목적

`ocr/extract_hwp_artifacts.py`에서 HWP 표 후보를 1차로 넓게 추출한 뒤,
실제로 활용 가능한 표를 단계적으로 좁혀 간다.

1차 분류:

- `cover`
- `toc`
- `sure_table`
- `review_needed`
- `uncertain`

2차 분류:

- `final_sure_table`
- `final_review_table`
- `discarded_table`

핵심은 1차에서 너무 많이 버리지 않고, 2차에서 `missing_signals`를 이용해
놓친 표를 다시 살리는 것이다.

## 1차 분류 기준

### cover

표지 성격이 강한 블록이다.

- `제안요청서`, `제안서`, `용역사업`, `사업명` 같은 표지형 키워드가 많다
- 기관명, 연도, 제목 위주다
- 데이터 표 헤더는 약하다

### toc

목차 성격이 강한 블록이다.

- `목차`, `차례`가 있다
- `Ⅰ.`, `Ⅱ.`, `1.`, `2.` 같은 번호 줄이 많다
- 섹션명 나열 중심이다

### sure_table

실제로 표일 가능성이 매우 높은 블록이다.

조건:

- `row_hint_count >= 5`
- `paragraph_count >= 4`
- `data_score >= 4`
- 헤더 그룹 3/3 매칭
- 설명형 bullet 블록이 아님

### review_needed

표일 가능성은 있지만 더 확인이 필요한 블록이다.

조건:

- `row_hint_count >= 3`
- `paragraph_count >= 3`
- `data_score >= 4`
- 헤더 그룹 2/3 매칭

### uncertain

위 규칙으로 확정하기 어려운 블록이다.

이 단계에서는 바로 버리기보다, 2차에서 다시 볼 후보로 둔다.

## 헤더 그룹 방식

단일 키워드 개수보다, 표 헤더 세트가 얼마나 맞는지를 본다.

예시 그룹:

- `구분`, `항목`, `내용`
- `구분`, `내용`, `비고`
- `요구사항`, `ID`, `명칭`
- `요구사항`, `분류`, `명칭`
- `평가`, `배점`, `기준`
- `평가항목`, `배점`, `평가기준`
- `일정`, `기간`, `비고`
- `예산`, `금액`, `합계`
- `수량`, `단가`, `금액`
- `구분`, `역할`, `비고`

한 블록 안에서 가장 많이 맞은 그룹의 개수를 `header_group_hits`로 사용한다.

- 3/3 일치: `sure_table` 후보
- 2/3 일치: `review_needed` 후보

## 설명형 블록 억제

아래 조건이면 설명형 본문이 강한 블록으로 보고 `sure_table`에서 제외한다.

- bullet 줄이 2개 이상
- 긴 줄이 2개 이상
- 헤더 신호는 약함

이 규칙을 두는 이유:

- `사업개요`
- `추진배경`
- `추진방안`

같은 설명형 블록이 표처럼 오인되는 걸 줄이기 위해서다.

## 2차 보정: missing_signals

1차 분류만으로는 박스형 표, 좌측 레이블형 표, 헤더가 일부 빠진 표를 놓칠 수 있다.
그래서 `missing_signals`를 계산해 2차 분류에 반영한다.

### weak_header_match

조건:

- `row_hint_count`는 높다
- `data_score`도 어느 정도 있다
- 그런데 `header_group_hits`는 낮다

의미:

- 실제 표 구조는 있는데 헤더가 일부 깨졌거나 빠졌을 가능성

### left_label_column_pattern

조건:

- 짧은 줄이 반복된다
- `추진목표`, `추진과제`, `대민사이트`, `BackOffice`, `추가사항`, `구분`, `역할` 같은 레이블형 키워드가 있다

의미:

- 좌측 카테고리 열이 있는 표일 가능성

### box_layout_like

조건:

- `row_hint_count`가 높다
- 화살표 또는 레이블형 키워드가 섞여 있다

의미:

- 일반 데이터 표는 아니지만, 박스형 레이아웃 표일 가능성

### explanatory_block

조건:

- bullet 중심
- 긴 설명문이 많음
- 헤더는 약함

의미:

- 2차에서도 살리지 않는 편이 좋은 설명형 블록

### promotable_review_table

조건:

- 1차에서 `review_needed`
- 구조는 충분히 강함

의미:

- 2차에서 `final_sure_table`로 승격할 수 있는 후보

## 2차 최종 분류

- `sure_table` -> `final_sure_table`
- `review_needed` + 승격 신호 있음 -> `final_sure_table`
- `review_needed` -> `final_review_table`
- `uncertain` + `missing_signals`가 강함 -> `final_review_table`
- 나머지 `uncertain` -> `discarded_table`

## 이 수정이 들어간 이유

1차 분류는 정밀도는 올려주지만, 아래 유형을 놓칠 수 있다.

- 박스형 표
- 좌측 레이블형 표
- 헤더 일부 누락 표

그래서 2차는 더 많이 버리는 단계가 아니라,
1차에서 놓친 유용한 표를 다시 끌어올리는 단계로 설계했다.

## 현재 기대 효과

- `사업개요`, `추진방안` 같은 설명형 블록은 덜 올라온다
- `요구사항`, `평가`, `배점`, `산출물` 표는 `sure_table`로 남기기 쉬워진다
- `추진목표 / 대민사이트 / BackOffice` 같은 박스형 표는 `missing_signals`로 다시 살릴 수 있다

## 샘플 비교 결과 정리

샘플 파일:

- 원본: `(사)벤처기업협회_2024년 벤처확인종합관리시스템 기능 고도화 용역사업 .hwp`
- 추출 결과: `ocr/sample_hwp_artifacts.json`

### 잘 잡힌 표의 공통점

현재 `final_sure_table`로 잘 들어간 표들은 아래 특징을 공통으로 가진다.

- 헤더가 명확하다
- 같은 행 구조가 반복된다
- 텍스트만으로도 `헤더 -> 값` 관계가 유지된다
- `header_group_hits`가 높다

대표 예:

- `요구사항 고유번호 / 요구사항 분류 / 요구사항 명칭`
- `평가부문 / 평가항목 / 평가기준 / 평가요소 / 배점기준`
- `산출물 종류 / 제출시기 / 수량 / 비고`

즉 현재 규칙은 `헤더형 데이터 표`에는 강하다.

### 놓친 표의 공통점

`discarded_table` 쪽을 보면, 실제로는 살릴 가치가 있는데도 버려진 표가 있다.

대표 유형:

- 박스형 표
- 좌측 레이블형 표
- 제목 블록과 본문 표가 분리된 표
- 설명이 긴 체크형 표

대표 예:

- `추진목표` 박스
- `추진체계 / 역할`
- `구 분 / 역 할`
- `구분 / 항목 / 적용계획/결과 / 미적용시 사유`
- `번호 / 보안약점 / 설명 / 비고`

이 표들은 텍스트 자체는 어느 정도 추출되지만, 아래 이유로 버려진다.

- 헤더 그룹 매칭이 약함
- `구 분`, `역 할`, `비 고`처럼 띄어쓰기 변형이 있음
- 좌측 레이블 열 구조가 일반 표처럼 안 보임
- 본문 설명이 길어서 `explanatory_block`으로 눌림
- 제목 블록과 본문이 분리되어 있음

즉 현재 규칙은 `레이아웃형 표`에는 약하다.

## 현재 판단

지금까지의 결과를 요약하면:

- HWP에서 쓸모 있는 표를 읽는 1차 추출은 성공했다
- 하지만 구조 판단이 약해서 버려진 데이터 표가 존재한다
- 특히 `헤더형 표`는 잘 잡고, `레이아웃형 표`는 놓치는 경향이 있다

## 다음 보강 방향

다음 단계에서는 전체 규칙을 바꾸기보다, 놓친 표 쪽 신호에 가중치를 더 주는 방향이 적절하다.

우선 보강 후보:

- `left_label_column_pattern`
- `box_layout_like`
- `weak_header_match`
- 띄어쓰기 정규화 후 헤더 재매칭

즉 현재 방향은:

- 잘 잡히는 표 규칙은 유지
- 놓친 표를 다시 살릴 수 있는 신호만 강화

## 2차 필터 보강 메모

이번 보강에서는 아래 두 가지를 추가했다.

### 1. 매칭용 정규화

`구 분`, `역 할`, `설 명`, `비 고`처럼 글자 사이 공백이 들어간 헤더 때문에
헤더 매칭이 약해지는 문제가 있었다.

그래서 비교용 텍스트를 정규화해서:

- 공백 제거
- 제어문자 제거

후에 다시 키워드와 헤더 그룹을 매칭하도록 수정했다.

이 변경으로 아래 같은 표가 잘 살아난다.

- `구 분 / 역 할`
- `번호 / 보안약점 / 설 명 / 비 고`
- `구 분 / 항 목 / 적용계획/결과`

### 2. 놓친 표 쪽 신호 강화

`missing_signals` 쪽에서 아래 패턴을 더 강하게 반영했다.

- `left_label_column_pattern`
- `box_layout_like`
- `weak_header_match`

특히 아래 유형을 살리기 위한 목적이다.

- `추진 / 목표`
- `추진체계 / 역할`
- `추진일정`
- 좌측 라벨 열이 있는 표
- 보안점검/적용계획 같은 긴 체크형 표

샘플 기준:

- 2차: `final_sure_table 143`, `final_review_table 19`, `discarded_table 55`

대표 변화:

- `table_index 8` -> `final_review_table`

- `table_index 12` -> `final_sure_table`
- `table_index 14` -> `final_review_table`
- `table_index 196` -> `final_sure_table`
- `table_index 200` -> `final_sure_table`
- `table_index 215` -> `final_sure_table`
- `table_index 216` -> `final_sure_table`

즉 현재 2차 필터는

- 헤더형 데이터 표는 그대로 유지하고
- 구조 때문에 놓치던 레이아웃형 표를 추가로 살리는 방향으로 동작한다.
